# HDAW HTML/Electron Frontend — Implementation Plan

## Architectural decisions (confirmed)

1. **Transport**: New WebSocket server in the engine (`src/frontend/`) that maps JSON-RPC methods **1:1 to the six abstract interfaces** in `src/common/` (`ProjectCommands`, `TransportCommands`, `AudioGraphCommands`, `ReadModel`, `PluginService`, `PluginParamService`). Engine runs as a headless child process spawned by Electron.
2. **Frontend**: React + TypeScript, built with Vite, packaged by Electron.
3. **Plugin editors**: Native popups owned by the engine process (`toggleFXEditor` via RPC) + generic HTML parameter sliders for parameter-only control.
4. **Phasing**: Thin vertical slice end-to-end first, then phase in the full surface.

## Why this design works with the existing code

- **`HDAW_lib` is already GUI-free** (`CMakeLists.txt:49-88`). It links `Qt6::Core/Network/HttpServer` + JUCE + SoundTouch, compiles `JUCE_PLUGINHOST_VST3=1`, and contains the entire engine, model, and MCP server. The headless frontend server links this lib and adds nothing Qt-Widgets-related.
- **The command interfaces are pure C++** (verified: every signature takes `std::string`/`int`/`double`/`float`/`bool`/POD structs — zero `juce::ValueTree` or `juce::String` leakage). They serialize to JSON trivially.
- **`McpJsonRpc.h`** (`parseLine`, `validateRequest`, `serializeResponse`) is reusable for WebSocket framing — JSON-RPC 2.0 over WS is the same line protocol as over stdio.
- **`AudioEngine` is already a root-tree `ValueTree::Listener`** (`AudioEngine.h:20,74-76`). The frontend watcher attaches the same way (`engine.getProjectModel().getTree().addListener(this)`) — the documented-safe pattern from AGENTS.md that survives `File→New` rebuilds.
- **Meter/playhead polling is already atomic-safe**: `LevelMeter` and `TransportManager` expose `std::atomic<float>`/atomics read by UI timers today. A 30 Hz server-side timer reading these atomics (no locks, no audio-thread contact) mirrors the existing `VUMeter` pattern exactly.
- **`--mcp-stdio` already proves headless engine operation** (`main.cpp:62-85`). The new `--headless` mode is the same shape with `QCoreApplication` (no `QApplication`) — `QWebSocketServer` doesn't need `Qt6::Widgets`.

## Wire protocol

JSON-RPC 2.0 over a single WebSocket. Three message classes:

**Request (client → server):**
```json
{"jsonrpc":"2.0","id":1,"method":"project.addTrack","params":{"name":"Lead"}}
```

**Response (server → client):**
```json
{"jsonrpc":"2.0","id":1,"result":{"trackIndex":3}}
```

**Notification (server → client, push):**
- `notify.treeChanged` — fired by the root-tree `ValueTree::Listener`; client re-fetches `read.snapshot()`. (Coarse-grained v1; fine-grained property diffs are a later optimization.)
- `notify.meters` — ~30 Hz: `{master:{l,r}, tracks:[{l,r},...]}`. Server reads atomics, no locks.
- `notify.transport` — ~30 Hz: `{currentTimeSeconds, isPlaying, isRecording, ...}`. Drives playhead + timecode.
- `notify.scanProgress` / `notify.exportProgress` — when relevant.

Method namespace mirrors the interfaces exactly: `project.*`, `transport.*`, `audioGraph.*`, `read.*`, `plugin.*`, `pluginParam.*`, `midi.*`. **The new router calls the command interfaces directly** — it does **not** mirror `McpTools.cpp`, which bypasses them (documented tech debt). MCP is left untouched for LLM clients.

## Phase 0 — Engine-side server foundation

**Goal**: a working headless WebSocket server that any WS client can drive end-to-end.

### New files

- `src/frontend/FrontendServer.h` / `.cpp` — `QObject` owning a `QWebSocketServer`. Constructor takes `AudioEngine&`; grabs the six interface pointers in `initialize()`. Methods: `start(quint16 port)`, `stop()`, `broadcastNotification(method, params)`.
- `src/frontend/FrontendRpc.h` — method-name constants + `QJsonObject` ↔ POD struct converters (one helper per `*Snapshot` struct; these are the only place that knows the JSON shape).
- `src/frontend/FrontendRouter.cpp` — the dispatch core. `QJsonValue dispatch(method, params, &err)` switches on method prefix and calls the matching interface method. Returns the result JSON. Main-thread-only (matches the project's single-thread rule).
- `src/frontend/FrontendTreeWatcher.h` / `.cpp` — a `juce::ValueTree::Listener` that attaches to `engine.getProjectModel().getTree()` in `initialize()` and calls `server->broadcastNotification("notify.treeChanged", {})` on any property/child change. (Debounced ~16 ms to coalesce burst edits.)
- `src/frontend/main_headless.cpp` — alternative entry, or extend `src/main.cpp` with `--headless [--port=N]` mode that builds `QCoreApplication`, `AudioEngine`, `FrontendServer`, runs `app.exec()`. (v1 reuses `HDAW.exe` with the flag; a dedicated `hdaw_headless` target is a later cleanup.)

### Modified files

- `CMakeLists.txt`:
  - Add `WebSockets` to `find_package(Qt6 ... COMPONENTS ...)`.
  - Add `Qt6::WebSockets` to `HDAW_lib`'s `target_link_libraries`.
  - Add the new `src/frontend/*.cpp` to `HDAW_lib`'s source list.
- `src/main.cpp`: add `--headless` and `--port=` parsing; construct `FrontendServer` instead of `MainWindow` when set.

### Tests (pilot, following the MCP test pattern)

- `tests/unit/frontend/frontend_server_test.cpp` — uses a `QWebSocket` client to `read.snapshot()`, `project.addTrack()`, `transport.play()`, asserts round-trips. Uses `QSignalSpy::wait()` for the async path; never `sleep()`.

### Exit criteria

A standalone script (or `wscat`) can: connect to `ws://127.0.0.1:8766`, call `read.snapshot()`, `project.addTrack({name:"X"})`, `transport.play()`, and observe `notify.treeChanged` + `notify.transport` push messages.

## Phase 1 — Thin vertical slice (proves the architecture)

**Goal**: an Electron app that spawns the engine, renders transport + tracks + mixer + piano roll + a minimal timeline, and round-trips real mutations. Validates the wire protocol, push channel, and React render performance before committing to the full surface.

### Electron + React scaffolding (new `frontend/` directory)

- `frontend/package.json` — Electron, Vite, React, TypeScript, Zustand (state), Konva or PixiJS (canvas, decided in Phase 2).
- `frontend/electron/main.ts` — spawns `build/Debug/HDAW.exe --headless --port=<N>`, waits for the WS port to accept, creates `BrowserWindow` loading the Vite dev server (dev) or packaged `dist/index.html` (prod). Handles child lifecycle: kill on `app.quit()`, restart-on-crash dialog.
- `frontend/electron/preload.ts` — exposes a thin `window.hdaw` RPC handle to the renderer.
- `frontend/vite.config.ts` — React plugin, dev server proxy.
- `frontend/tsconfig.json`.

### Renderer core

- `frontend/src/rpc/client.ts` — JSON-RPC WebSocket client: `call(method, params)` returns a Promise; subscription API for `notify.*`. Auto-reconnect with backoff.
- `frontend/src/store/projectStore.ts` — Zustand store. On `notify.treeChanged`, re-fetches `read.snapshot()` and replaces state. Selectors per area.
- `frontend/src/store/transportStore.ts` — updated by `notify.transport` at 30 Hz.
- `frontend/src/store/meterStore.ts` — updated by `notify.meters` at 30 Hz; drives mixer VUs and master meter.
- `frontend/src/theme.ts` — port of `src/ui/Theme.h` colors to CSS variables.

### Thin-slice UI components

- `App.tsx` — top-level layout (menu bar placeholder, transport bar, main splitter, status bar).
- `components/TransportBar.tsx` — play/stop/rewind/record/loop, BPM, time-sig, timecode (bound to `transportStore`). Proves button → RPC → push round-trip.
- `components/Mixer.tsx` + `MixerStrip.tsx` — fader, pan, mute/solo, VU bars (bound to `meterStore`). Proves continuous-control + 30 Hz meter rendering.
- `components/PianoRoll.tsx` + `NoteGrid.tsx` — the most HTML-friendly custom paint (per the exploration: pure 2D grid, trivial hit-testing). Uses `<canvas>`. Proves canvas editing works.
- `components/TimelineMinimal.tsx` — static clip rectangles (no drag yet), playhead driven by `transportStore`. Proves the timeline concept renders.
- `components/TrackHeaders.tsx` — name/volume/pan/mute/solo. Proves per-track editing.

### Exit criteria

User can launch the Electron app, see the default project, add/remove tracks, drag mixer faders with live VU response, play/stop with moving playhead, and edit MIDI notes in the piano roll — all with the engine running headless as a child process. Latency budget: <50 ms click-to-visible for discrete ops; smooth 60 FPS for meters/playhead.

## Phase 2 — Full timeline (the hard part)

**Goal**: timeline parity with the Qt `QGraphicsScene`.

### Canvas engine

- Choose **Konva** or **PixiJS** (decide after a perf spike in Phase 1 — Konva for ergonomics, Pixi if 200-clip scenes stutter). The timeline is the perf-critical surface.
- `components/timeline/TimelineCanvas.tsx` — viewport with horizontal/vertical scroll, zoom (1–200 pps).
- `components/timeline/ClipView.tsx` — clip body, color, selection outline, fade triangles.
- `components/timeline/AudioClipWaveform.tsx` — needs Phase 2b.
- `components/timeline/MidiClipPreview.tsx` — mini note preview.
- `components/timeline/TrackLanes.tsx` — vertical track layout, track heights.
- `components/timeline/Ruler.tsx` — beat/bar/second ticks, tempo points from `read.getTempoPoints()`.
- `components/timeline/Playhead.tsx` — bound to `transportStore` at 30 Hz.
- `components/timeline/LoopMarkers.tsx` + `MarkerFlags.tsx`.

### Interaction layer

- `lib/timeline/interaction.ts` — port of `TimelineInteraction.h` DragMode enum {None, Move, TrimLeft, TrimRight, FadeIn, FadeOut, RubberBand, Duplicate}. Snap-to-grid, multi-select (rubber-band + additive), cross-track drag with continuous Y-tracking (the v0.9.1 fix). All drags batch into a single `project.beginTransaction(...)` / `endTransaction()` pair so they're one undo step (the AGENTS.md undo-coalescing rule).
- `lib/timeline/scrollSync.ts` — three-pane sync (headers / ruler / scene), the documented pitfall.

### Phase 2b — Waveform peaks endpoint (new engine RPC)

The current waveforms use `juce::AudioThumbnail` server-side. Add:

- `read.getWaveformPeaks({clipId, samplesPerPixel})` → `{min: number[], max: number[]}` (or a binary little-endian Float32 over WS for throughput). Backed by `ProjectPool`'s existing thumbnail cache.
- Client caches peaks keyed on `(clipId, samplesPerPixel)` — mirrors the existing `QPixmap` cache key in `AudioClipItem`.

### Exit criteria

Timeline parity with Qt: drag clips across tracks smoothly, rubber-band select, trim/fade handles, zoom/scroll, waveforms render at all zoom levels, playhead tracks. Cross-reference the AGENTS.md "Clip drag Y-tracking" pitfall — the continuous-follow behavior must reproduce.

## Phase 3 — Bottom panel tabs

Port each tab. Most are low-complexity point/line editors over beat/time axes.

- **FX Chain** (`FXChain.tsx`, `FXSlotRow.tsx`): slot list, bypass, drag-reorder, generic param sliders driven by `pluginParam.getParams` / `pluginParam.setParam`. "Edit" button calls `audioGraph.toggleFXEditor` → engine pops native window.
- **Automation** (`AutomationLane.tsx`): point canvas, add/remove/drag points, lane add/remove menu (driven by `read.getAutomatableParams`), full clipboard keybindings (Ctrl+C/X/V/D/A, Delete, Escape — port verbatim from `AutomationLaneWidget`).
- **Audio Clip Editor** (`AudioClipEditor.tsx`, `GainEnvelope.tsx`): detailed waveform (reuse Phase 2b peaks), gain slider, fades, loop, offset/duration, timestretch controls (Src BPM / Stretch combo / ratio / Fit-to-Loop), slicing buttons, region clipboard. Largest single component — split into subcomponents.
- **Modulation** (`Modulation.tsx`): LFO panels (waveform, rate, sync, depth, bipolar, phase, target). Note: `addLfo` has no return value — re-query `read.getModulationLfos(trackIndex)` after add to discover the new index (documented quirk).
- **Step Sequencer** (`StepSequencer.tsx`): 8×16 toggle grid.

### Exit criteria

All five bottom-panel tabs reach feature parity with Qt, including undo-coalesced drags and the automation clipboard.

## Phase 4 — Surrounding UI, dialogs, plugin popups

- **Menu bar** (`MenuBar.tsx`): File/Edit/Transport/Track/View/Tools/Help with the documented shortcuts (Space=Play, R=Record, Ctrl+Z=Undo, etc.). Use Electron's native `Menu` for platform-correct shortcuts.
- **Toolbar**: snap toggle + division, zoom controls, grid-type combo, follow-playhead, metronome, count-in, default-clip-length.
- **Status bar** (`StatusBar.tsx`): BPM, time-sig, sample rate, selected track, MIDI device, recording state, selection count.
- **Project pool browser** (`ProjectPoolBrowser.tsx`): file tree + project pool + audio preview + drag-drop import. Drag-drop from OS uses HTML5 DnD → `project.addAudioClip`.
- **Dialogs** (Electron `BrowserWindow` or in-app modal): Export, Preferences (audio device, sample rate, buffer, MCP/frontend port, plugin scan paths, defaults), Plugin Scanner (progress + blacklist), Phrase Generator, Startup, About.
- **Plugin editor popups**: native windows owned by the engine. RPC `audioGraph.toggleFXEditor(trackIndex, slotIndex)` → engine calls `TrackFXSlot::showEditor()`. No Electron code needed beyond the button. Documented as the v1 tradeoff.

### Exit criteria

Full GUI parity. Every Qt feature has an HTML equivalent or a documented native-popup fallback.

## Phase 5 — Cut over

- Flip the default: launching `HDAW.exe` (or a new `HDAW_Electron.exe` wrapper) starts Electron + headless engine.
- Keep the Qt GUI available behind `--legacy-qt` for one release as a safety net.
- Add a `docs/superpowers/specs/` design doc and an AGENTS.md section documenting the new architecture (the frontend server, the wire protocol, the headless mode).
- After stabilization, delete `src/ui/*` (except `PluginEditorWindow.cpp`, which `HDAW_lib` still needs for native plugin popups) and drop `Qt6::Widgets` from the build.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Timeline canvas perf with 200+ clips | Phase 1 perf spike before committing; PixiJS fallback; viewport culling (the existing `NoteGridWidget::paintEvent` cull pattern ports directly) |
| WebSocket throughput for waveform peaks at high zoom | Binary Float32 frames instead of JSON; client-side cache keyed on (clipId, samplesPerPixel) |
| Burst edits flood `notify.treeChanged` | Debounce ~16 ms in `FrontendTreeWatcher` (coalesce); client re-fetches snapshot once |
| Native plugin windows feel disconnected from Electron | Documented v1 tradeoff (user-approved); generic HTML params cover the rest |
| `--mcp-http-port` and new `--port=` could collide | Separate default ports (MCP HTTP 8765, frontend WS 8766) |
| Child process crash takes down the UI | Electron main detects child exit, shows restart dialog, offers "save log" |
| Two parallel mutation paths (MCP tools vs command interfaces) drifting further | New frontend server uses **only** the command interfaces; MCP left as-is for LLM clients. Future work: migrate `McpTools.cpp` onto the command interfaces too (documented in AGENTS.md) |

## Out of scope for v1 (deferred)

- Pixel-streamed plugin editors (native popup is the v1 path).
- Realtime stretch-during-playback without rebuild (already out of scope in the Qt version).
- Persisting rendered stretch buffers in `.hdaw` (re-rendered from props on load — existing behavior).
- Fine-grained per-property tree diffs (coarse `notify.treeChanged` + snapshot re-fetch is the v1 path).
- Migrating `McpTools.cpp` onto the command interfaces (separate tech-debt cleanup).
