## Problem

ctrl+drag copy and alt+drag paint are "erratic and keep regressing." Root-cause analysis found **two independent bugs**, both of which must be fixed for durable stability:

### Root cause #1 — broken return-value parsing (the silent killer)
The C++ backend returns a **bare integer** for `project.duplicateClip` and `project.createGhostClip` (`FrontendRouter.cpp:188-189`: `return { false, c.duplicateClip(i) }` → serialized as the raw `result` field, confirmed via `FrontendRpc.h:52` `DispatchResult.payload` + `FrontendServer.cpp:195` `sendResponse(id, result.payload)`).

But the frontend parses them as an object: `if (r && typeof r === "object" && "clipId" in r)` (`useTimelineDrag.ts:110, 161, 188`). Since `typeof 2 === "object"` is `false`, **the new-id capture never executes**. Result: `newIds` stays empty → selection never swaps to the duplicates → mouseup's move targets the *original* clips → `syncSnapshot` then shows duplicates sitting where `duplicateClip` placed them (`start+duration`) while originals get moved. This is exactly the reported symptom. It also means the paint loop's `paintedClipIds` never advances, so its rollback path is dead code.

**Why it "keeps regressing":** the existing test (`useTimelineDrag.test.ts:170`) mocks `mockResolvedValue({ clipId: 2 })` — a shape the real backend never produces — so the test passes while production fails. Any fix verified only by tests looked fine then broke in the real app.

### Root cause #2 — timing race on eager-duplicate
Even if parsing were fixed, the duplicate fires inside an `async` IIFE on the first `mousemove` (lines 177–202) that reassigns `dragSelectedIdsRef.current = newIds` only *after* the RPC resolves. `handleMouseUp` reads that ref synchronously. So a fast release (before the RPC resolves) targets the original clips; a slow release works. Same gesture, different outcome → "erratic."

---

## Fix (confirmed direction with user)

**A. Lazy on mouseup for ctrl+drag and ghost-clone** — defer all RPC work to a single synchronous decision point (mouseup). During the drag we only track the preview (already exists via `dragPreviewStyle`, which works and does not depend on `duplicated`/`ghostDuplicated`). On mouseup we know the final target → one transaction: `beginTransaction` → `duplicateClip`/`createGhostClip` per id (parsed as **bare int**) → `moveClipWithOverlap` to the computed target → `endTransaction` → swap `dragSelectedIdsRef` + selection to the new ids → `syncSnapshot` once. No async reassignment during the drag → the race window is eliminated entirely. This matches the stable pattern normal (non-copy) drag already uses and that pitfall #3 endorses.

**B. Alt-drag paint → existing atomic `project.paintClips` RPC** — replace the client-side per-tile `duplicateClip`+`moveClipWithOverlap` loop with one call to `project.paintClips({ sourceClipIds, originBeat, spacing, targetTrackIndex, count })`, which already wraps itself in a transaction and applies correct relative-offset math (`AudioEngineCommands_GhostPaint.cpp:87,149`, positioning `originBeat + (tile+1)*spacing + relativeOffset`). It returns `number[]` (bare). During the drag we show `paintTiles` (pending previews) and the `+N` badge as today; commit happens once on mouseup. This deletes ~30 lines of fragile client logic and removes per-tile flicker/dead rollback.

### Files to change

1. **`frontend/src/hooks/useTimelineDrag.ts`** (the rewrite)
   - `handleMouseMove`: for `isDuplicate` and `isGhostClone`, remove the eager async-duplicate branches. Just update `mouseX/mouseY` (so the preview tracks the mouse — currently it freezes during the async window). Keep `paintRepeat` branch only to update `mouseX/mouseY` + recompute desired count for the live preview; do NOT issue per-tile RPCs during the drag anymore.
   - `handleMouseUp`: central commit point. Three branches:
     - normal move: unchanged (already correct).
     - `isDuplicate`: `beginTransaction` → for each original id, `await rpc.call("project.duplicateClip", {clipId:id})` parsed as `const newId = typeof r === "number" ? r : -1`; collect `Map<origId,newId>`; then `moveClipWithOverlap` each new id to `targetStart + (orig.startBeat - dragStartBeat)`, same delta track; `endTransaction`; set selection + `dragSelectedIdsRef` to new ids; `syncSnapshot` once. No optimistic mid-drag placement.
     - `isGhostClone`: same shape using `createGhostClip(sourceId, newStart, newTrackIdx)`, bare-int parse.
     - `paintRepeat`: compute final `count` from mouse position; if `count > 0` call `project.paintClips` once (bare-int array parse); swap `dragSelectedIdsRef` + selection to returned ids; `syncSnapshot` once.
   - Drop now-unused `duplicated`/`ghostDuplicated` latch logic. `paintSpacing`/`paintOriginBeat` are computed at mousedown (keep).
   - Keep the existing store-read pattern (the uncommitted fix at line 251) and apply the same `useProjectStore.getState().snapshot?.clips` discipline in the new mouseup branches (pitfalls #1/#4).

2. **`frontend/src/utils/timelineConstants.ts`** — the `DragState` type: drop `duplicated?` and `ghostDuplicated?` (internal-only, confirmed no external reader). Keep `isDuplicate`/`isGhostClone` (still needed to branch on mouseup) and all `paint*` fields (read by `TimelineMinimal.tsx:646-647`).

3. **`frontend/src/components/TimelineMinimal.tsx`** — no logic change expected (it reads only `paintRepeat`, `paintOriginBeat`, `paintSpacing`, `paintedClipIds.length`, `dragSelectedIdsRef.current.has`, and the derived hook outputs). Will re-verify after the rewrite; only touch if a derived output shape changed.

4. **`frontend/src/hooks/useTimelineDrag.test.ts`** — fix the mocks to return the **real** backend shapes:
   - `duplicateClip` / `createGhostClip` → bare `number` (e.g. `mockResolvedValue(2)`), not `{clipId:2}`.
   - `paintClips` → `number[]` (e.g. `mockResolvedValue([3,4,5])`).
   - Update assertions: ctrl+drag now duplicates on mouseup (not first mousemove), so the `duplicateClip` expectation moves after `handleMouseUp`. Add a regression test asserting that after a ctrl+drag, selection contains the **new** id and the new clip is at the drop position (this is the assertion that would have caught root cause #1).
   - Keep the "alt-drag waits for RPC" intent but reframe: paint RPC fires on mouseup.

### What is explicitly NOT changing
- Backend C++: no changes. `paintClips` already exists and is correct; `duplicateClip`/`createGhostClip`/`moveClipWithOverlap` semantics are fine.
- Normal (non-copy) drag: unchanged (already stable).
- Ghost semantics on ctrl+drag of a ghost clip: preserved as-is (duplicate keeps `isGhost`/`ghostSourceId` from `createCopy()`). Out of scope to re-root; flagged for a future decision only.
- `handleClipMouseDown` signature incl. the `forcePaintRepeat` 5th arg: unchanged (used at `TimelineMinimal.tsx:622`).

### Verification
- `cd frontend && npm test` (Vitest) — existing + updated/added tests, especially the new bare-int regression test.
- `cd frontend && npm run build` — typecheck (DragState field removal must not break callers).
- Manual: the user will exercise ctrl+drag (fast and slow release), alt+drag paint, ghost-clone (ctrl+shift), and the loop-paint handle in the running app, since the regression was timing/IPC-sensitive and only fully reproduces against the real engine.

### Why this is durable (not another point-fix)
- Removes root cause #1 by parsing the real wire shape (and fixes the test that masked it).
- Removes root cause #2 by collapsing copy/paint into a single commit-on-release phase, eliminating the async-during-drag that created the timing window.
- Eliminates the duplicated client-side paint logic in favor of the existing atomic backend RPC.
- Aligns copy/paint with the already-stable normal-drag pattern, so there is one mental model for all clip drags.
