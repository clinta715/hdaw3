# Pitch-Preserving Timestretch for Audio Clips

## Approach (one-line)

Render a stretched copy of each clip's audio **off the audio thread** (SoundTouch in a worker, mirroring `ExportManager`), cache the result, and have `ClipSourceProcessor::prepareToPlay` adopt it during the normal `rebuildRoutingGraph()` path — so the realtime `processBlock` stays a cheap 1:1 copy, just reading from a second buffer selected by one atomic. The stretch *intent* lives in the ValueTree; the stretched *audio* is regenerated from it.

The realtime audio path gains **one atomic load and one pointer select per block** — no allocation, no locks, no resampling on the audio thread. All stretch policy/decisions are message-thread + worker.

## New ValueTree properties (`src/model/ProjectModel.h` IDs block, ~line 44)

All on the CLIP node, audio clips only:

| ID | Type | Default | Meaning |
|----|------|---------|---------|
| `sourceBpm` | double | `0.0` | Musical tempo of the source file. `0` = unknown (TempoMatch disabled). |
| `stretchMode` | int | `0` | `0=Off`, `1=TempoMatch`, `2=ManualRatio` |
| `stretchRatio` | double | `1.0` | Effective **time** ratio vs. original source (`targetDuration/sourceDuration`). TempoMatch derives it; ManualRatio is user-set. |
| `sourceDuration` | double | (set at import) | Original source length in seconds; stored once so re-renders don't re-open the file. |

`ProjectModel::createAudioClip` (`ProjectModel.cpp:189-205`) sets these defaults. No existing IDs collide (confirmed).

**Semantics once stretched:** the clip's existing `duration` = `sourceDuration * stretchRatio` (timeline-visible length); `offset`/`gain`/`fade` window into the stretched buffer exactly as today. MIDI clips are untouched (they follow tempo natively).

## 1. `StretchRenderer` — SoundTouch wrapper + worker (`src/engine/StretchRenderer.{h,cpp}`)

Mirrors `ExportManager` (`src/engine/ExportManager.{h,cpp}`) exactly:

- Members: `std::thread renderThread`, `std::atomic<bool> active{false}`, `std::atomic<bool> cancelFlag{false}`, `std::function<void(float)> onProgress`, `std::function<void(bool,juce::HeapBlock<int>[2],int64_t)> onComplete`.
- `startRender(sourceFile, timeRatio, sampleRate)`: refuses if `active`; joins prior thread; spawns worker.
- Worker: decode source via `formatManager.createReaderFor` → `juce::AudioBuffer<float>`; configure SoundTouch (`setTempoChange`, `setPitchChange(0)` for pitch-preserving); push blocks through `putSamples`/`receiveSamples`; convert float→int(`/32768`) into two `HeapBlock<int>` matching `ClipSourceProcessor`'s format; poll `cancelFlag` per block; invoke `onComplete` (on worker thread) + `active=false`.
- `cancel()` sets the flag. Destructor: `cancel()` + `join()` (same as `ExportManager.cpp:7-12`).

Output heapblocks match `ClipSourceProcessor::preloadedData` (int, 2-ch) so adoption is a pointer move + memcpy, no conversion.

## 2. `StretchCache` (`src/engine/StretchCache.{h,cpp}`)

Owned by `AudioEngine` (next to `ProjectPool`). Avoids re-rendering on every `rebuildRoutingGraph`.

```cpp
struct Entry {
    double ratio; double sampleRate; int clipID;
    juce::HeapBlock<int> data[2]; int64_t length;
    std::atomic<bool> ready{false};
};
// key: (clipID, ratio, sampleRate)
void requestRender(int clipID, const juce::String& sourceFile, double ratio, double sr,
                   std::function<void()> onReady);   // no-op if matching entry ready
const Entry* lookup(int clipID, double ratio, double sr);  // for prepareToPlay
```

`onReady` is the completion callback (marshalled to message thread inside `requestRender` via `QMetaObject::invokeMethod(receiver, …, Qt::QueuedConnection)`) → calls `engine.getMainProcessor()->rebuildRoutingGraph()`. That rebuild is what swaps the stretched buffer into the playing clip safely (under `graphLock`).

`prepareToPlay` (message thread, during rebuild) calls `lookup()`; if `ready`, adopts the buffer and sets `activeBuffer=1`; else falls back to the original `preloadedData` (`activeBuffer=0`). No deadlock: worker never touches `graphLock`.

## 3. `ClipSourceProcessor` realtime change (`src/engine/ClipSourceProcessor.h`)

Minimal, realtime-safe:
- New members: `juce::HeapBlock<int> stretchedData[2]`, `int64_t stretchedLength{0}`, `std::atomic<int> activeBuffer{0}` (0=original, 1=stretched), `int clipID{-1}` (set by `RoutingManager::rebuildClipsForTrack`).
- `prepareToPlay`: after existing preload, query `StretchCache::lookup(clipID, ratio, sr)`; on hit, fill `stretchedData` + set `activeBuffer=1`.
- `processBlock`: **one** `activeBuffer.load(relaxed)` at the top; pick `src` pointer + length; the existing read loop is otherwise unchanged:

```cpp
const int buf = activeBuffer.load(std::memory_order_acquire);
const auto& src  = (buf == 1) ? stretchedData : preloadedData;
const int64_t len = (buf == 1) ? stretchedLength : preloadedLength;
int64_t sourceSample = offsetSamples + clipLocalSample;   // unchanged
// bounds/loop checks now use `len`; copy loop reads src[ch][...]
```

`RoutingManager::rebuildClipsForTrack` (`RoutingManager.cpp:339-372`) reads the new stretch props from the ValueTree and passes `clipID` + the resolved ratio into the processor. **No new SPSC paramID is needed** — stretch is decided at rebuild time, not per-sample (unlike volume/pan). This is the key simplification.

## 4. Import path — tempo-match on import

Two sites: `src/engine/AudioImport.cpp:39-64` and `src/ui/TimelineView.cpp:920-941`. Both already open a `reader` before `createAudioClip`. Insert between:

1. **Read BPM from metadata** (first `getMetadataValues()` use in the codebase): check WAV `bext`/`INFO` keys and ID3v2 `TBPM`; parse best-effort. Falls back to `0.0`.
2. If pref `autoTempoMatchOnImport` is on **and** a BPM was found: set `sourceBpm`, `stretchMode=TempoMatch`, `stretchRatio=sourceBpm/projectBpm`, `duration=sourceDuration*stretchRatio` on the clip node.
3. If no BPM: `stretchMode=Off`; the user can set `sourceBpm` in the editor later.
4. The ValueTree listener (§6) picks up the new props and kicks off the background render; the clip plays original until the render completes, then rebuild swaps it in. No import-time blocking for the stretch.

## 5. `AudioClipEditorWidget` UI (`src/ui/AudioClipEditorWidget.cpp:27-153`)

Insert a "Stretch" group into the control bar (there's already an `addStretch()` gap at line 78), gated to audio clips:
- **Source BPM** `QDoubleSpinBox` (0–400, "0 = unknown") → `setClipSourceBpm`.
- **Mode** `QComboBox`: Off / Tempo Match / Manual Ratio → `setClipStretchMode`.
- **Ratio** `QDoubleSpinBox` (0.25–4.0, step 0.01) → `setClipStretchRatio` (only enabled in Manual mode).
- **Fit to Loop** `QPushButton` → `fitClipToLoop(clipId)`.

`loadClip` reads the four new props and populates widgets (with the existing `settingUi` guard to avoid feedback). `fitClipToLoop` is disabled when the loop region is empty (`loopEnd <= loopStart`).

**Fit-to-loop semantics:** stretch the *entire source* to span exactly the loop region — `stretchRatio = (loopEnd−loopStart)/sourceDuration`, `stretchMode=ManualRatio`, `duration=loopEnd−loopStart`, `offset=0`. One-shot, deterministic.

## 6. Follow-project-tempo (`src/engine/AudioEngine.cpp`)

Extend the existing CLIP-property handler (`AudioEngine.cpp:297-330`). For `stretchMode`/`stretchRatio`/`sourceBpm` changes: do **not** push an SPSC update (it's not RT-parametric); instead call a new `AudioEngine::requestClipStretchUpdate(trackIdx, clipIdx)` that reads the clip's props and calls `StretchCache::requestRender`.

In the existing project-tempo handler (`AudioEngine.cpp:219-225`, after `transportManager.setBPM`): iterate all clips with `stretchMode==TempoMatch`, recompute `stretchRatio = sourceBpm/newBpm`, write it (undoable), and `requestClipStretchUpdate` for each. Each completion triggers its own rebuild. For many clips this is a burst of renders — acceptable (they're parallel-ish, no UI freeze); a batch "render-all-then-one-rebuild" is noted as an optimization.

## Phasing (so this stays reviewable)

- **Phase 1 (end-to-end value):** properties, `StretchRenderer`+SoundTouch build, `StretchCache`, `ClipSourceProcessor` RT change, `AudioClipEditorWidget` UI (**Manual Ratio + Fit to Loop + Source BPM**), `ProjectCommands`, tests. *Delivers timestretch you can drive by hand.*
- **Phase 2:** **Tempo-match on import** (metadata read + pref + import-path edits).
- **Phase 3:** **Follow project tempo** (AudioEngine tempo-change listener, batch re-render).

Phases build on Phase 1 and can land separately. Phase 1 alone is a usable, testable feature.

## Out of scope (noted, not built)
- Realtime/stretch-during-playback without rebuild (would need SPSC + lock-free buffer swap — deferred; changing stretch mid-playback does a brief silence blip, same as clip add today).
- Persisting rendered buffers in the `.hdaw` file (re-rendered from props on load; acceptable since renders are backgrounded).
- Per-clip "preview original vs stretched" A/B (nice-to-have UI, deferred).
- MCP tools (`set_clip_stretch`, `fit_clip_to_loop`) — straightforward follow-up consistent with the existing tool surface.
- Rubber Band swap-in (the interface makes this a one-file change later).
