# HTML GUI Review — Findings & Implementation Plan

A comprehensive review of the React/Electron frontend (`frontend/src/`) and its JSON-RPC contract with the C++ engine (`src/frontend/FrontendRouter.cpp`, `src/frontend/FrontendServer.cpp`). Findings are organized by category, then the plan lists fixes in priority order.

### C. Performance

| # | Location | Issue |
|---|---|---|
| C1 | `FrontendServer.cpp:172-175` | **30 Hz `notify.transport` broadcast is unconditional** — every connected client receives a full `TransportSnapshot` ~30×/sec even when idle and nothing has changed. Compare to `notify.meters` (expected to change continuously). |
| C2 | `TransportBar.tsx:71-82` | **Polls `syncSnapshot` + `syncDirtyFlag` every 250 ms** while playing/recording — redundant alongside the `notify.treeChanged` push (already debounced to 16 ms in `FrontendTreeWatcher`). |
| C3 | `FrontendServer.cpp:148` | `broadcastNotification` is **not thread-safe** despite the "Safe to call from any thread" comment in `FrontendServer.h:43`. Iterates `clients_` (QSet) with no lock; only safe because all current callers are on the main thread. A future background caller races against `onNewConnection`/`disconnected`. |
| C4 | `FrontendRouter.cpp:478-559` | `read.getWaveformPeaks` blocks the dispatch thread with synchronous disk reads. Re-decode every call, no cache. Long files = multi-second stall of the WebSocket handler. |
| C5 | `WaveformCanvas.tsx:11` | `peaksCache` lives forever and **never invalidates** when a clip source changes (e.g. after timestretch render). Stale waveforms. |

## Part 2 — Implementation Plan

One combined plan, in priority order (bugs → perf → quality → aesthetic). Build + smoke-test after each group.

### Group 1 — Fix the dead backend paths (A1, A2, A3, A4)

These four are the highest-impact: features that visibly exist in the UI but produce nothing.

1. **`ExportManager` RPC + progress notification**
   - Add an `export` namespace to `FrontendRouter.cpp::dispatch` (new `dispatchExport` helper) with method `export.audio({ outputPath, format, bitDepth })`. Delegate to the existing `ExportManager` (already used by the MCP tool `export_audio` in `McpExportTool.cpp:31`).
   - Make the export run on a worker thread (mirror the MCP path's `notifyFromBackground`).
   - **Emit `notify.exportProgress`** (not `notifications/progress`) from the export worker via `FrontendServer::broadcastNotification`. Wire `notify::ExportProgress` (already declared in `FrontendRpc.h:43`).
   - Decide: keep the MCP `export_audio` tool path separately (it has its own progress stream and dry-run semantics) or refactor both to call one shared helper. **Recommendation: refactor to one shared helper** in `src/engine/` or `src/frontend/` that both paths call.

2. **`transport.record` alias**
   - Add `transport.record` to `dispatchTransport` as an alias that toggles: if `isRecording()` calls `stopRecording()`, else `startRecording()`. Matches the UI semantics in `TransportBar.tsx:111` and the `R` shortcut. (Alternatively, change the frontend to call `startRecording`/`stopRecording` — but the alias is one line and matches user intent better.)

3. **`project.setClipGainEnvelope`**
   - Add `setClipGainEnvelope` to `dispatchProject`: takes `{ clipId, points: [{time, gain}, ...]`. Implementation: `clearGainEnvelope` then iterate `addGainEnvelopePoint` per point, wrapped in a `beginTransaction`/`endTransaction` so it's one undo step. Add the method to `ProjectCommands` interface and `AudioEngineCommands` impl.

### Group 2 — Fix the field-name mismatches (A5, A6, A10)

Pick the canonical field names and align both sides. These are one-line fixes each but require touching both frontend and backend (or just frontend, depending on choice).

4. **`pluginParam.getParams` field name**
   - Backend returns `index` (`FrontendRouter.cpp:423`); frontend reads `paramIndex` (`FXChain.tsx:278`).
   - **Recommendation: change backend to emit `paramIndex`** for consistency with `AutomatableParamSnapshot` (which already uses `paramIndex`, `rpc/types.ts:99`) and with the write side (`pluginParam.setParam` takes `paramIndex`, `FrontendRouter.cpp:441`). One name across all plugin-param surfaces.

5. **`addFxSlot` param names**
   - Frontend sends `{ trackIndex, fxType, slotIndex, pluginId }`; router reads `{ trackIndex, type, position, pluginId }` (`FrontendRouter.cpp:210-213`).
   - **Recommendation: change router to accept both `type`/`fxType` and `position`/`slotIndex`** (read either, fall back). Zero-risk, no frontend change needed, future-proof.

6. **`TrackSnapshot` type honesty (A10)**
   - Remove `type`, `meterL`, `meterR` from the TS interface (they're never populated; meters come via `notify.meters`). Add a code comment noting where meter data actually arrives.

### Group 3 — Fix the visibly-broken widgets (A7, A8, B3, B10, D4)

7. **`CCLane.tsx` — actually draw the CC points**
   - Add a `useEffect` that calls `read.getNotes`-equivalent for CC data. **Issue:** there's currently no `read.getCcPoints` RPC. Either add one to `dispatchRead` (cheap; mirrors `getAutomationPoints`) or expose CC points on the existing clip snapshot.
   - Draw existing points as connected line segments; highlight hover/click target.

8. **`StepSequencer.tsx` — wire to backend or remove**
   - **Recommendation: wire it.** On toggle, call `project.addNote`/`project.removeNote` for the cell's pitch×step (pitch from `BASE_NOTE + (ROWS-1-row)`, startBeat = `col * 1/16`, durationBeats = `1/16`). On mount/clip-change, populate `steps` from the existing notes via `read.getNotes`. Keep grid snap to 1/16.
   - Wrap multi-step changes (clear, fill) in `beginTransaction`/`endTransaction`.

9. **Canvas CSS-variable bug (B3)**
   - Replace `ctx.fillStyle = "var(--bg-widget)"` / `ctx.strokeStyle = "var(--accent)"` in `GainEnvelopeEditor.tsx` with **resolved values from `getComputedStyle(document.documentElement).getPropertyValue("--bg-widget")`**, cached once per draw. Same fix needed anywhere else Canvas reads theme tokens (audit `AutomationLaneCanvas`, `WaveformCanvas`, `VelocityLane`, `NoteGrid`, `CCLane` once it draws).

10. **`MixerStrip` color bug (B10/D4)**
    - Use `colorStr(track.color)` (already defined in `TrackHeaders.tsx:56`) — extract to a shared `theme.ts` helper and import in both. Same fix for `Mixer.tsx:30` and the App.tsx floating-button inline colors.

### Group 4 — Performance (C1, C2, C4, C5)

11. **Suppress idle transport broadcasts (C1)**
    - In `FrontendServer` (or wherever the 30 Hz timer fires), compare the new `TransportSnapshot` to the last-sent one (sample-by-sample diff of `currentTimeSeconds` rounded, plus bool flips for `isPlaying`/`isLooping`/`isRecording`). Skip broadcast if unchanged.
    - Keep broadcasting when `isPlaying` (playhead moves), but stop when fully idle.

12. **Drop the 250 ms snapshot poll (C2)**
    - The `notify.treeChanged` push (16 ms debounce) already covers it. Remove the `setInterval` in `TransportBar.tsx:71-82`, keep only `syncDirtyFlag` if dirty-flag isn't pushed (it isn't — consider adding `notify.dirtyChanged`).

13. **Async + cached waveform peaks (C4)**
    - Move the peak-decode loop off the dispatch thread. Options: (a) return a "pending" response immediately and push `notify.waveformPeaksReady` when done, or (b) run a worker pool keyed by `(clipId, sourceMtime, numBins)`.
    - **Recommendation: shared cache layer** in `src/engine/WaveformPeakCache.{h,cpp}` keyed by `(filePath, mtime, numBins)` — both `FrontendRouter::read.getWaveformPeaks` and `McpTools::get_waveform_peaks` call into it (also resolves B5).
    - Frontend `WaveformCanvas.tsx` cache invalidation (C5): key on `sourceFile + sourceMtime` (needs the mtime on `ClipSnapshot`, or just re-fetch when the snapshot's `stretchRatio` changes).

14. **`broadcastNotification` thread safety (C3)**
    - Either (a) add a `QMutex` around the `clients_` iteration, or (b) `QMetaObject::invokeMethod(this, ..., Qt::QueuedConnection)` from any non-main-thread caller. **Recommendation (b)** — keep the main-thread fast path lock-free; only the queued hop pays.

### Group 5 — Code quality (B1, B2, B4, B5, B7, B11, B12)

15. **Consolidate keyboard handling (B1)**
    - Single source of truth in `useKeyboardShortcuts.ts`. Remove the `keydown` listener in `TimelineMinimal.tsx:595-707` (move timeline-specific keys into the hook with a guard like `if (activeBottomTab !== "timeline") return`). Same for `FileMenu.tsx:168-182`.
    - Use a priority/early-return pattern so `Ctrl+N` doesn't fire twice.

16. **De-duplicate `GainEnvelopeEditor` draw (B2)**
    - Extract a `drawEnvelope(ctx, w, h, pts, theme)` function. Both `handleWindowMouseMove` and the `useEffect` call it. ~80 lines → ~25.

17. **`FileBrowser` directory-load dedup (B4)**
    - One `loadDirectory(path)` function, two call sites.

18. **Waveform peaks shared helper (B5)** — covered by item 13.

19. **`peaksCache` invalidation + scoping (B7)**
    - Move `peaksCache` into a `WaveformCache` class instance (or a React context). Invalidate on `sourceFile` or `stretchRatio` change.

20. **Batched deletions (A12)**
    - `automationStore.removePoints` and `NoteGrid.deleteSelected`: wrap the per-item RPC loop in `beginTransaction`/`endTransaction`. Better: add `project.removeNotes`/`project.removeAutomationPoints` batch methods that take an array.

21. **`PluginManagerDialog` batched blacklist fetch (A11)**
    - Add `plugin.getBlacklistStatus` returning a map of `{ pluginID: bool }` for all plugins in one call. Frontend makes one round-trip.

### Group 6 — Aesthetic / theme (D1, D2, D3, D5, D6, D7)

22. **Theme-ify `FileBrowser.css` (D1)**
    - Replace all hardcoded hex with CSS custom properties from `theme.ts`. Verify against existing tokens; add new ones if needed.

23. **Replace inline-style colors (D2, D3)**
    - `App.tsx:74-86` browser toggle button, `Mixer.tsx:30` master color. Move to CSS classes; use theme vars.

24. **Consolidate tiny CSS files (D5)**
    - Merge `CCLane.css`, `GainEnvelopeEditor.css` (etc.) into the nearest larger neighbor (e.g. `PianoRoll.css` for `CCLane`/`VelocityLane`/`GainEnvelopeEditor`; `Mixer.css` for `MixerStrip`). Or keep separate but ensure they actually use theme tokens. **Recommendation: keep separate files** (component-scoped CSS is a feature) but audit for theme-token compliance.

25. **Global context-menu styles (D7)**
    - Extract a shared `.hdaw-context-menu` class in `App.css` (or a new `ContextMenu.css`); components add only their specific items.

26. **Error UI for failed RPC calls (D6)**
    - Add a `useRpcErrorHandler` hook or wrap `rpc.call` so failures surface a transient toast (status bar) with the method name and error message. Critical for catching future silent-failure bugs.
    - **Recommendation: minimal first cut** — a `notifyStore` Zustand store + a `<Toaster>` component mounted at the App root. Failed calls push `{method, message, ts}`; Toaster renders + auto-dismisses after 4 s.

## Out of scope (noted, not addressed)

- The `dist/` and `dist-electron/` directories may be checked in (verify against `.gitignore`) — out of scope for this review.
- WebSocket server has no origin/auth check (relies on loopback binding). Acceptable for v0.9.x; document as a v1 follow-up.
- MCP HTTP transport's notification path is a no-op (`McpTransportHttp.cpp:103-104`) — only relevant for MCP clients, not the HTML frontend.
- `ProjectModel::getTransportTree` metronome/count-in/time-sig not in `TransportSnapshot` — handled in frontend's `transportExtrasStore` by design; the `TODO` at `transportExtrasStore.ts:16` is intentional deferral.
