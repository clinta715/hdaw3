# Plan: Fix 2 UI bugs + Moderate decoupling backsliding cleanup

## Context (from the audit)

The audit found:
- **2 real bugs**: a truncated dB label and an unguarded paint clip-rect in `TrackHeaderWidget`.
- **Decoupling backsliding**: v0.6.0 added `ProjectCommands`/`ReadModel`/etc. interfaces and widgets store the pointers, but hot paths (paint, VU timers) still call `engine.getProjectModel()` / `engine.getMainProcessor()` directly. Two widgets (`MixerStripWidget`, `TrackHeaderWidget`) re-read the whole model in `paintEvent` — the exact "paint-during-read" anti-pattern AGENTS.md flags as a documented follow-up.

Key facts that shape the design (verified during planning):
- `readModel->getTrackMeter()`/`getMasterMeter()` are **pure atomic loads** — a strict win over `engine.getMainProcessor()->getTrack(i)->getMeter()`. Safe in timers.
- `readModel->getTrack(i)` allocates a `std::string` per call and re-scans ValueTrees — **NOT** a free swap for paint loops. The correct fix is `ValueTree::Listener` caching into existing member fields (both widgets already have cached members + `setXxx()` setters that are never wired up).
- **AGENTS.md pitfall**: child-tree listeners are orphaned by `ProjectSerializer::createNew`/`load` (`removeAllChildren`). The safe pattern (used by `TimelineView:37`) is to attach to the **root tree** and filter by `hasType()`. This plan uses that pattern.
- `ProjectCommands` has **no** `beginTransaction()` — all migrated widgets still use `engine.getProjectModel().getUndoManager()` for fine-grained edits. Per the Moderate scope, we leave undo-path coupling as documented debt (the Aggressive option would add the API).
- Both widgets already construct the 4 interface pointers and store cached display state; the gap is purely the missing listener wiring + leftover reads.

## Scope decision (confirmed with user)
- **Moderate**: clean wins (meter reads, `getTrackCount`, tempo accessor) + fix the paint-during-read anti-pattern via `ValueTree::Listener` caching. Leave event-handler undo calls + FXSlotRow as documented debt.
- **Add Case 3 accessor** (`getAutomatableParams`) now. Defer Case 2 (FXSlotRow live param polling — needs a new `PluginParamService`).

## Part A — The two bugs (do first, smallest + isolated)

### A1. `TrackHeaderWidget.cpp:399` — dB label truncates toward zero
**Root cause**: `static_cast<int>(negative_double)` truncates toward zero, so -3.7 dB → "-3dB".
**Fix**: round before casting.
```cpp
int db = static_cast<int>(std::round(20.0 * std::log10((std::max)(vol, 0.001f))));
```
Add `<cmath>` include guard if missing (likely already present via the log10 usage). One-line change.

### A2. `TrackHeaderWidget.cpp:297` — paint clip-rect can go negative
**Root cause**: `height() - rulerHeight` is negative if the widget is momentarily sized < 30px during layout, collapsing all track rows (the documented startup "tracks missing" symptom).
**Fix**: clamp the height at the clip-rect site (defensive, matches AGENTS.md guidance):
```cpp
int clipH = (std::max)(0, height() - static_cast<int>(rulerHeight));
painter.setClipRect(0, static_cast<int>(rulerHeight), w, clipH);
```
One-line change. `sizeHint()` already tries to prevent the small-widget case; this is the belt-and-suspenders guard AGENTS.md explicitly recommends.

**Verification**: `cmake --build build --config Debug` compiles; visual check that track headers + volume labels render correctly at startup and after File→New. No new test needed for these (UI paint details aren't in the gtest suite; the build + manual smoke check is the established bar per AGENTS.md).

## Part C — `MixerStripWidget`: become a `ValueTree::Listener` (the documented follow-up)

This widget already has cached `name`/`volume`/`pan`/`muted`/`soloed` members and `setXxx()` setters — they're just never driven by a listener, so `paintEvent` re-reads the model. The fix is listener wiring + removing the paint-time read.

### C1. Header changes (`MixerStripWidget.h`)
- Make the class inherit `juce::ValueTree::Listener` (multiple inheritance alongside `QWidget` — already done this way by `TimelineView`).
- Add overrides: `valueTreePropertyChanged`, `valueTreeChildAdded`, `valueTreeChildRemoved`.
- Add private members: `juce::ValueTree trackTree;` (the watched track node).

### C2. Constructor (`MixerStripWidget.cpp:13`)
- Keep the existing initial read from the ValueTree (lines 30-40) — it seeds the cache. Also store `trackTree = trackList.getChild(trackIndex);`.
- Attach listener to the **root tree** (NOT the child) per AGENTS.md pitfall: `engine.getProjectModel().getTree().addListener(this);`

### C3. Listener callbacks (`MixerStripWidget.cpp`, new methods)
- `valueTreePropertyChanged(tree, property)`: if `tree == trackTree` (compare by `juce::ValueTree` equality, which is reference-based), switch on property and call the existing `setXxx()` setters (which call `update()`). Properties: `IDs::name`→`setTrackName`, `volume`→`setVolume`, `pan`→`setPan`, `isMuted`→`setMuted`, `isSoloed`→`setSoloed`.
- `valueTreeChildAdded` / `valueTreeChildRemoved`: if a TRACK is added/removed and indices shift, the strip's `trackIndex` may now point at a different track. Simplest correct behavior: emit `trackDeleted()` if our track's identifier no longer matches (the parent `MixerWidget::rebuild()` already tears down and rebuilds strips on track add/remove, so this is belt-and-suspenders). Keep minimal.

### C4. `paintEvent` (`MixerStripWidget.cpp:106`)
- **Remove** lines 108-117 (the `getTrackListTree()` read block). Paint uses only the cached members, which are now kept current by the listener.

### C5. `updateVU()` (`MixerStripWidget.cpp:87`)
- Replace `engine.getMainProcessor()->getTrack(trackIndex)->getMeter()` with `readModel->getTrackMeter(trackIndex)`. This is a pure atomic-load read — safe and cheaper. Null-track guard becomes the `MeterSnapshot` zero-default.

**Verification**: build; manual smoke check that dragging a fader repaints the strip and that changing volume from the track header (or MCP) updates the mixer strip live. No new gtest (widget behavior).

## Part D — `TrackHeaderWidget`: same listener pattern + meter migration

Same shape as Part C, applied to the per-track `TrackHeader` struct cache. This widget is larger (975 lines) and iterates all tracks in paint, so this is the highest-value change.

### D1. Header (`TrackHeaderWidget.h`)
- Inherit `juce::ValueTree::Listener`.
- Add the three listener overrides.

### D2. Constructor — attach to **root tree**: `engine.getProjectModel().getTree().addListener(this);`

### D3. `rebuild()` — when the track list is reconstructed, the `tracks` vector is already repopulated here. Ensure each `TrackHeader` caches `volValue`/`panValue`/`isMuted`/`isSoloed` from the ValueTree at build time (some of this may already happen; audit and complete it).

### D4. Listener callback `valueTreePropertyChanged(tree, property)`
- If `tree.hasType(IDs::TRACK)`: find the matching track in `tracks` by identity (walk `tracks` and compare the stored track tree, OR resolve the track index via `getProjectModel().getTrackListTree().indexOf(tree)`). Then update the cached field on the `TrackHeader`:
  - `IDs::volume` → `header.volValue`
  - `IDs::pan` → `header.panValue`
  - `IDs::isMuted` → `header.isMuted`
  - `IDs::isSoloed` → `header.isSoloed`
  - `IDs::name` → (used in paint via `tree.getProperty`; either cache it or leave the name read in paint since names are cheap and rarely change — prefer caching for consistency)
  - Call `update()` after the mutation.
- **Important**: the listener fires on the message thread (same as paint), so no thread-safety concern. But mutations originating from *this widget's own mouse handlers* (e.g. dragging volume) will also trigger the listener — that's fine and keeps the cache in sync; the drag already updates the ValueTree.

### D5. `paintEvent` (`TrackHeaderWidget.cpp:279-408`)
- Remove the per-track `tree.getProperty(IDs::volume/pan/isMuted/isSoloed)` reads; read from the cached `header.volValue`/etc. instead.
- This also **removes the `getTrackListTree()` call at line 285** from the paint path. The track count still comes from `tracks.size()` (already the source of truth after `rebuild()`).

### D6. `updateVU()` (`TrackHeaderWidget.cpp:104`)
- Replace `engine.getMainProcessor()->getTrack(h.trackIndex)->getMeter()` with `readModel->getTrackMeter(h.trackIndex)`. Atomic-load swap.

### D7. `sizeHint()` (`TrackHeaderWidget.cpp:262`)
- Replace `engine.getProjectModel().getTrackListTree().getNumChildren()` with `readModel->getTrackCount()`. Cheap (one top-level scan, no string alloc).

**Verification**: build; smoke-test that (a) VU meters still animate, (b) volume/pan/mute/solo drags update the header, (c) changes from MCP or the mixer reflect in the header live (proving the listener works), (d) File→New doesn't orphan the listener (root-tree attachment survives). No new gtest.
